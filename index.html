站外影響｜AEO（AI Engine Optimization）

當用戶直接問 ChatGPT / Google AI 買什麼，AI 直接推薦商品，電商平台可能被繞過（有 AI 摘要的頁面跳出率 +10%）。威脅：自然流量下降。機會：商品資料越結構化，越容易被 AI 引擎引用推薦。

站內：① 商品頁結構化資料（Schema.org）② 商品描述品質提升 ③ 用戶評論和 Q&A 資料

站外（Off-site Content Strategy）：與外部 blogger、媒體合作；善用 Yahoo 奇摩新聞做交叉引用，站外高權威內容被 AI 引用機率更高。

夥伴關係（AI Partnership）：主動與 Google Shopping Graph、OpenAI、Perplexity 建立官方資料合作，直接提交商品 Feed。Yahoo 的護城河：7-ELEVEN 取貨資訊、OPENPOINT 回饋、到貨時效——這些結構化資訊是競品沒有的。

1

Entity Recognition 建立
建立電商知識圖譜需要時間，越早啟動越好。精準 Query 用戶 CVR 潛力最高。
3
Embedding 導入（分詞升級）
與 Entity Recognition 並行。為 Hybrid Search 做語意匹配準備。
第二批｜4–6 週後（依賴第一批基礎）
4
Intent Classification 基礎建立
依賴 Entity Recognition 有初步覆蓋率後才準確。Q2 所有工作的前置條件。
5
Hybrid Search 上線
依賴 Embedding 完成。詞彙搜尋 + 向量搜尋並行，用 RRF 合併，降低 Null Rate 最直接路徑。
6
Re-ranking 加入
依賴 Hybrid Search 穩定後才加。Cross-encoder 對前 50–100 個商品深度精排。
核心邏輯
為什麼 Query 先於 UI/UX？
UI/UX 做好了，Query 不準，流量就 lost。這是 VoiceTube 的直接教訓：第一版優化覆蓋率提升但 CVR 下降，因為分詞邏輯出問題，UI 帶進來的流量在 Query 層全部流失。反過來說，Query 地基穩了，UI/UX 的每一個改善才能帶來真實的轉換提升。
Q1 因果鏈
Spell Correction → 降低 Null Rate，讓有意圖的用戶進得來
Entity Recognition → 讓精準搜尋用戶找得到正確商品
Intent Classification → 讓系統理解意圖，為 Q2 做地基
Hybrid Search → 語意覆蓋，進一步降低 Null Rate
Re-ranking → 前排結果更精準，CTR 和 CVR 提升
中期（Q2）｜UI/UX + Campaign 整合 + RAG 嵌入

核心邏輯
為什麼 Q2 才做 UI/UX？
UI/UX 的效果取決於 Query 的品質。Q1 的 Intent Classification 和 Entity Recognition 完成之前，Zero State 設計再漂亮，用戶搜一個長尾詞出來一堆不相關商品，體驗還是爛的。

RAG 為什麼 Q2 才做？
RAG 的品質上限取決於 Retrieval 的品質。底層搜尋爛，LLM 再強也無法把爛結果變好。Q1 的 Hybrid Search 先到位，RAG 才有意義。

Campaign 整合為什麼分兩步？
第一步快速實作（Campaign Banner），不改 Ranking 邏輯，立刻讓 Search 和活動有連結。第二步系統化（Query→Campaign Mapping），需要 Intent Classification 成熟後才能精準定義觸發條件。
長期｜差異化護城河，競品無法複製
✕ 收合
三個差異化方向
1
統一集團資源整合（7-ELEVEN × Search）
搜尋結果商品卡片整合超商取貨資訊、到貨時效、OPENPOINT 回饋。酷澎物流標籤的 Yahoo 版，但競品複製不了。
2
OPENPOINT 個人化 Embedding
線下消費行為數據（7-ELEVEN、家樂福）整合後，個人化深度超越純電商競品。知道用戶線下買什麼，比只看線上瀏覽行為更能預測購買意圖。
3
Conversational AI 個人化成熟
數據夠豐富後才值得建立獨立 AI 購物助理入口。沒有個人化數據的 Conversational AI 跟通用 LLM 差不多，做出來反而傷品牌信任。
核心邏輯
為什麼這是護城河？
Q1、Q2 是縮短與競品的差距，任何平台理論上都可以做。長期的差異化在於統一集團的線下資源——7-ELEVEN 取貨、OPENPOINT 點數、到貨時效——這些是其他電商平台沒有的結構化資訊。

一旦這些資訊整合進 Search 結果和 AI 個人化，蝦皮、momo、酷澎都複製不了，因為他們沒有這個生態系。

時間點的判斷：
個人化 Embedding 需要 OPENPOINT 跨平台數據整合成熟才有意義，不應該提前投資。資料不夠豐富時做出來的個人化效果跟全站熱銷差不多，浪費工程資源。
終態｜Unified Intent-aware Search
✕ 收合
架構
用戶輸入（單一入口）
    ↓
Intent Router（AI 分類）
    ↓
交易型 → AI-enhanced Search → 傳統 SERP
探索型 → Conversational AI   → 對話式推薦卡片
混合型 → 兩者並行         → 混合 SERP

三個子問題：

① Intent Router 怎麼判斷
短 Query（「Nike 球鞋」）傾向交易型；長句（「我想找適合夏天戶外活動的防曬」）傾向探索型。持續用點擊和 CVR 數據訓練模型。

② 兩個引擎怎麼並行
各自回傳候選結果，Merge Layer 做 RRF 合併。Router 不是非此即彼，而是動態權重分配。延遲控制是關鍵：並行不串行，設定 Timeout。

③ UI 層怎麼統一呈現
彈性 SERP 模板：上方可插入 AI 生成摘要，下方仍是商品列表。用戶感受不到切換，只感受到「有人幫我整理過」。類比：Google AI Overview 就是這個概念。
核心邏輯
為什麼是終態而不是現在就做？
Intent Router 需要大量真實 Query 數據訓練，準確率不夠就會路由錯誤，反而傷搜尋體驗。UI 混合呈現需要前端架構重設計，投入大。三個前提都要到位：

前提條件
Q1 底層品質到位（Hybrid Search、Re-ranking）
Q2 RAG 在缺口場景驗證有效
長期個人化數據夠豐富

一句話總結：
AI 和搜尋的整合不是「要不要用 AI」的問題，而是「在哪個階段、用哪種方式讓 AI 介入」。短期修復 Query 地基，中期在缺口插入 RAG，長期用統一集團資源建護城河，終態是讓用戶感受不到兩個系統的存在，只感受到「搜尋幫我找到了我要的東西」。
